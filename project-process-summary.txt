Handwriting Formula Lab 작업 과정 정리
====================================

1. 초기 아이디어
---------------
처음 목표는 개인의 한글 필기체를 인식해서 하나의 글씨체로 만들고, 사용자가 문장을 입력하면 그 문장이 개인 필기체처럼 PDF로 변환되는 웹서비스를 만드는 것이었다.

초기 사용자 여정은 다음과 같았다.

1. 개인 손글씨 샘플을 가이드라인에 따라 입력한다.
2. 입력된 손글씨 샘플을 바탕으로 AI가 개인 한글 폰트를 생성한다.
3. 사용자가 문장을 입력한다.
4. PDF를 생성한다.

배포 방식은 Vercel을 목표로 정했고, GitHub 저장소는 Yonsei-BIT/sprint-2를 사용하기로 했다.


2. 첫 번째 프로토타입: 한글 손글씨 PDF 변환
-----------------------------------------
처음에는 정적 웹사이트 형태로 Handwriting Font Lab을 만들었다.

구현한 기능은 다음과 같았다.

- 패드/펜슬로 손글씨 샘플 입력
- 한글 자모, 숫자, 자주 쓰는 문장 샘플 수집
- 브라우저에서 개인 글꼴 프로필 생성
- 입력 문장 미리보기
- PDF 생성
- 변형도, 기울기, 글자 간격, 줄 간격 조절

하지만 이 버전은 실제 폰트 생성이 아니라, 저장한 손글씨 이미지를 합성하거나 브라우저 기본 글꼴에 스타일을 입히는 수준이었다. 그래서 사용자가 보기에는 “내 손글씨 폰트”라기보다 일반 글꼴에 효과를 넣은 것처럼 보였다.


3. 한글 버전에서 발견한 문제
---------------------------
한글 버전에서는 다음 문제가 계속 발생했다.

- 사용자가 쓴 글씨가 그대로 폰트화되는 것이 아니라 샘플 이미지처럼 출력되는 문제
- 자모 커버리지가 100%여도 실제 자음/모음 조합이 자연스럽지 않은 문제
- 한글 초성, 중성, 종성 위치가 맞지 않아 일반적인 글씨체처럼 보이지 않는 문제
- 기본 고딕체 골격을 쓰면 손글씨처럼 보이지 않는 문제
- 한글 완성형 글자를 모두 쓰게 할 수 없어 사용자 부담이 너무 큰 문제

이 과정에서 한글은 자모 조합 구조 때문에 MVP로 자연스러운 개인 폰트를 만들기 어렵다는 결론을 얻었다.


4. 실제 TTF 생성 MVP로 전환
--------------------------
그 다음에는 브라우저 합성 방식이 아니라 실제 .ttf 파일을 생성하는 구조로 바꾸었다.

새 구조는 다음과 같았다.

- 패드 입력 시 stroke 좌표, 시간, 압력 저장
- 샘플을 JSON + PNG로 export
- Python 서버에서 stroke를 정규화
- 입력된 샘플을 실제 TrueType glyph로 변환
- 생성된 .ttf를 브라우저에서 FontFace로 로드
- PDF에는 생성된 폰트를 적용

이를 위해 server.py를 추가했고, 외부 라이브러리 없이 Python 표준 라이브러리만으로 최소 TrueType 폰트 생성기를 구현했다.

이 버전부터는 실제로 /api/font에 stroke 데이터를 보내면 .ttf 바이너리를 생성하고, 생성된 파일이 TrueType Font로 인식되는 것을 확인했다.


5. 최소 샘플 기반 한글 합성 시도
-------------------------------
사용자가 모든 한글 글자를 쓸 수 없기 때문에, 필수 자음/모음/숫자만 입력하고 서버가 필요한 한글 글자를 합성하는 방식도 시도했다.

구현한 내용은 다음과 같았다.

- 필수 샘플 41개로 축소
- 한글 초성, 중성, 종성 분해
- 사용자가 입력한 문장 targetText를 서버로 전달
- 서버가 자모 샘플을 초성/중성/종성 위치에 배치해 완성형 한글 glyph 생성

하지만 결과적으로 한글은 자모 배치가 매우 민감하고, 자연스러운 필기체를 만들기에는 여전히 어렵다고 판단했다.


6. 영어 + 수식 버전으로 방향 전환
--------------------------------
한글 자모 조합 문제를 피하고 더 현실적인 결과를 만들기 위해 서비스를 영어 + 수식 버전으로 전환했다.

새 목표는 다음과 같다.

파일을 업로드하거나 텍스트/수식을 입력하면, 영어와 수식이 사용자의 필기체 스타일로 변환되어 PDF로 출력되는 사이트를 만든다.

새 사이트 이름은 Handwriting Formula Lab으로 변경했다.

사용자 샘플은 다음 문자들로 구성했다.

- 영어 소문자 a-z
- 영어 대문자 A-Z
- 숫자 0-9
- 수식 기호 +, -, =, ×, ÷, ±, ≠, <, >, ≤, ≥
- 괄호류 (, ), [, ], {, }
- 기타 수식/구두점 /, \, ^, _, ., ,, :, ;, ', "
- √, ∑, ∫, ∞, π, θ, Δ

총 96개 샘플을 입력하는 구조로 바꾸었다.


7. 파일 입력 기능 추가
---------------------
사용자가 직접 글을 입력하는 것뿐 아니라 파일도 넣을 수 있도록 했다.

구현한 기능은 다음과 같다.

- TXT 파일 업로드 시 브라우저에서 바로 텍스트 읽기
- PDF 파일 업로드 시 /api/extract로 서버에 전달
- Python 서버가 PDF에서 텍스트를 추출해 입력란에 반영

사용자가 제공한 Chp.10-11.pdf 파일로 PDF 추출 API 응답을 테스트했다. 다만 외부 PDF 라이브러리 없이 구현한 MVP라, PDF 텍스트 추출은 일부 글자가 깨지거나 레이아웃이 완벽하지 않을 수 있다.


8. 입력 UX 개선
--------------
패드와 Apple Pencil로 샘플을 입력하면서 여러 사용성 문제가 발견되었다.

개선한 내용은 다음과 같다.

- 패드에서도 접속 가능하도록 서버를 0.0.0.0:3000에 바인딩
- 접속 주소: http://172.24.146.182:3000/
- 캔버스에 가이드 박스 추가
- 가이드 박스 크기를 여러 차례 조정
- 너무 큰 가이드가 평소 필기 습관을 왜곡해서 1/3 크기로 축소
- 윗선/중심/아랫선 텍스트 라벨 제거
- 기준선과 박스 선을 연하게 조정
- Apple Pencil 입력 중 주변 글씨가 선택되지 않도록 user-select, touch-action, selectstart, dragstart 제어
- 펜 두께를 더 얇게 조정
- 빠른 필기에서 좌표가 빠지지 않도록 getCoalescedEvents 사용
- PointerEvent가 없는 환경에서는 touch/mouse fallback 사용


9. 출력 UX 개선
--------------
초기 출력 결과는 글씨가 너무 크고, 편지지/줄 노트처럼 보여서 실제 문서 느낌과 달랐다.

개선한 내용은 다음과 같다.

- 결과 배경을 편지지가 아닌 흰 백지로 변경
- 최종 글씨 크기를 48px에서 16px로 줄여 일반 문서의 12pt에 가깝게 조정
- 기본 변형도를 낮춰 글씨가 덜 흔들리게 변경
- 기본 기울기를 0으로 변경
- 글자 간격과 줄 간격을 문서형 출력에 맞게 조정


10. 영어/수식 폰트 자연스러움 개선
---------------------------------
영어+수식 버전에서도 처음에는 모든 glyph가 같은 정사각형 크기로 정규화되어 어색했다.

발견한 문제는 다음과 같다.

- 대문자와 소문자 크기가 같음
- 괄호와 숫자 사이 간격이 너무 벌어짐
- 문자, 숫자, 연산자, 괄호가 모두 같은 폭으로 취급됨
- 수식처럼 보이지 않고 글자가 등간격으로 나열됨

이를 개선하기 위해 다음을 적용했다.

- 소문자는 x-height 기준으로 작게 정규화
- b, d, f, h, k, l, t 같은 ascender 소문자는 더 높게 정규화
- g, j, p, q, y 같은 descender 소문자는 아래로 내려가게 정규화
- 대문자는 cap-height 기준으로 크게 정규화
- 숫자는 대문자보다 살짝 낮게 정규화
- 괄호, 구두점은 좁은 폭과 좁은 advance width 적용
- 연산자 주변 간격은 문자 간격보다 좁게 처리
- 프론트 미리보기는 고정 자간 대신 measureText 기반으로 배치
- 괄호, 구두점, 지수 기호, 연산자 주변에는 별도 pair spacing 적용


11. 샘플 저장 및 수정 기능
-------------------------
매번 96개 샘플을 다시 쓰는 것은 불편하기 때문에 브라우저 저장 기능을 추가했다.

구현한 내용은 다음과 같다.

- localStorage에 샘플 자동 저장
- 같은 브라우저로 다시 접속하면 샘플 자동 복원
- 이전/다음 버튼으로 특정 glyph에 돌아가면 저장된 샘플이 캔버스에 다시 표시됨
- 현재 샘플 수정 버튼으로 해당 글자 샘플만 삭제
- 샘플 저장/덮어쓰기 버튼으로 수정된 샘플 저장
- 샘플을 한 번 입력한 뒤에는 바로 텍스트/파일을 넣고 폰트 적용 가능


12. 현재 시스템 구조
-------------------
현재 구현된 구조는 다음과 같다.

Frontend:
- index.html
- styles.css
- app.js

주요 기능:
- 샘플 입력 캔버스
- Apple Pencil/터치/마우스 stroke 수집
- 샘플 localStorage 저장
- TXT/PDF 파일 업로드
- 텍스트 입력
- TTF 생성 요청
- 생성된 폰트 미리보기
- PDF 생성

Backend:
- server.py

주요 API:
- POST /api/font
  - stroke 데이터를 받아 최소 TrueType 폰트 생성

- POST /api/extract
  - TXT/PDF 파일에서 텍스트 추출


13. 현재 한계
------------
현재 버전은 실제 AI 모델이 학습하는 방식은 아니다. 사용자가 입력한 stroke를 정규화해서 glyph로 만들고, 이를 TTF로 패키징하는 MVP다.

현재 한계는 다음과 같다.

- 샘플이 없는 문자는 기본 폰트로 대체됨
- PDF 텍스트 추출은 외부 라이브러리 없이 구현되어 완벽하지 않음
- 수식의 위첨자/아래첨자 구조를 진짜 수식 레이아웃으로 파싱하지는 않음
- 필기체 연결성이나 문맥 기반 변형은 아직 없음
- AI 보간/생성 모델은 아직 붙어 있지 않음


14. 다음 단계 제안
-----------------
더 자연스러운 서비스로 만들려면 다음 단계가 필요하다.

1. PDF 텍스트 추출 고도화
   - PyMuPDF, pdfminer.six 같은 라이브러리 사용

2. 수식 파싱 고도화
   - LaTeX 또는 MathML 기반으로 위첨자/아래첨자/분수/적분 표현 처리

3. 샘플 요구량 최적화
   - 모든 문자가 아니라 대표 glyph만 받고, 비슷한 문자끼리 보간

4. AI 보간 모델 추가
   - 사용자의 일부 샘플을 보고 없는 글자/기호를 생성

5. 폰트 품질 개선
   - 곡선 smoothing
   - stroke outline 품질 개선
   - kerning pair 추가
   - glyph별 advance width 세밀화

6. 배포 구조 정리
   - Vercel frontend
   - Python backend는 Render/Railway/Fly.io/Modal 등에 배포


15. 현재 실행 방법
-----------------
로컬 서버 실행:

python3 server.py

패드에서 접속:

http://172.24.146.182:3000/

현재 서버는 Python 표준 라이브러리만 사용하며, 외부 패키지 설치 없이 동작한다.
